# 上传队列单图标签覆盖设计

## 目标

调整上传页的标签编辑模型，使上传队列中的待上传图片除了支持删除外，还支持单独修改标签，同时保留顶部区域作为这一批图片的默认批量标签设置。

本次设计需要同时满足以下目标：

- 顶部标签设置继续作为整批图片的默认值。
- 队列中每张待上传图片可以单独加标签，也可以单独取消某个批量默认标签。
- 顶部批量标签改动后，队列中每张图片的标签状态实时刷新。
- 单图标签修改与顶部批量默认值共存，互不粗暴覆盖。
- 上传时每张图片提交自己的最终标签集合，而不是统一复用顶部默认标签。

## 当前问题

现有上传页实现中，顶部批量标签是唯一的标签来源，上传时会把 `defaultTagIds` 和 `newTagNames` 直接发给每一张图片：

- 队列项本身没有独立标签状态。
- 队列项只支持删除，不支持单图标签编辑。
- 顶部标签虽然能表达“整批默认值”，但无法表达“某一张图片例外”。
- 如果直接把完整标签数组复制到每张图片上，后续顶部批量变更和单图变更会互相覆盖，状态容易失真。

## 范围

本次只覆盖上传页中“待上传 / 上传失败可重试”的队列项标签编辑能力。

包含：

- 顶部批量标签与队列项单图标签交互
- 队列项最终标签计算
- 上传请求改为按单图提交最终标签
- 与上述逻辑直接相关的测试

不包含：

- 单图描述字段编辑
- 上传完成后的图库详情页标签编辑
- 上传接口协议的大改
- 额外引入新的后端持久化中间态

## 核心决策

### 1. 顶部标签是“默认批量值”，不是最终真值

顶部区域继续维护：

- `defaultTagIds`
- `newTagNames`

它们表示这一批图片默认应当带上的标签，而不是队列中每张图片已经最终确认的完整标签集合。

### 2. 队列项采用“覆盖层”而不是“完整快照”

每个队列项除了文件信息和上传状态外，还维护单图标签覆盖层：

- `addedTagIds`
- `removedDefaultTagIds`
- `addedTagNames`
- `removedDefaultTagNames`

这样做的原因：

- 顶部默认标签变更后，队列项可以自动反映默认值变化。
- 单图改动只记录“差异”，不会因为顶部再次改动而被整段重写。
- 上传时可以稳定计算出该图片自己的最终标签集合。

### 3. 单图标签语义采用“可加可减”

用户已确认单图编辑采用以下语义：

- 可以给该图片额外增加批量默认值之外的标签。
- 可以把顶部批量默认标签只在这一张图片上取消。

这意味着队列项必须同时支持：

- “单图新增”
- “单图覆盖取消默认”

## 数据模型

### 顶部批量状态

继续保留现有状态：

- `defaultTagIds: string[]`
- `newTagNames: string`

其中 `newTagNames` 继续通过统一的解析函数转成去重后的字符串数组。

### 队列项状态

`UploadQueueItem` 需要扩展以下字段：

- `addedTagIds: string[]`
- `removedDefaultTagIds: string[]`
- `addedTagNames: string[]`
- `removedDefaultTagNames: string[]`

这些字段只描述该图片相对顶部默认值的偏移，不直接保存整张图片的最终标签全量快照。

### 最终标签计算

新增一个纯函数 helper，负责基于顶部默认值和单图覆盖层计算最终结果。

计算规则：

- `effectiveTagIds = defaultTagIds - removedDefaultTagIds + addedTagIds`
- `effectiveTagNames = defaultTagNames - removedDefaultTagNames + addedTagNames`
- 全部结果在输出前统一去重
- 字符串标签在比较前做 `trim`
- 空字符串不参与结果

这个 helper 同时作为页面渲染和上传 payload 的统一真值来源，避免 UI 展示与实际上传不一致。

## 交互设计

### 顶部区域

顶部区域仍然表示这一批图片的默认标签设置：

- 点击已有标签，表示给整批图片默认加上或取消该标签。
- 输入新标签名，表示给整批图片默认附加这些新标签。

顶部状态变化后，队列中所有待上传图片的标签显示需要实时跟随刷新。

### 队列项区域

处于 `ready` 或 `failed` 状态的队列项，除了删除按钮外，增加单图标签编辑区。

队列项标签编辑需支持两类操作：

- 对已有标签进行单图加/单图取消默认
- 对新标签名进行单图新增/单图取消默认

队列项 UI 至少需要清楚表达三种来源：

- 来自顶部默认值
- 当前图片额外新增
- 当前图片单独取消的默认标签

重点不是装饰，而是让用户一眼看出这张图最终会带哪些标签，以及哪些是该图片的例外。

### 顶部与单图联动规则

联动规则如下：

1. 如果某个标签只来自顶部默认值，则它会实时出现在每个队列项的最终结果中。
2. 如果某张图片把某个默认标签单独取消，则顶部后续的其他标签变更不会抹掉这个取消状态。
3. 如果顶部本身取消了某个默认标签，则该标签应从所有图片的默认来源中消失；相关“取消默认”覆盖记录可以在计算中自然失效。
4. 如果某张图片额外新增了标签，则顶部改动不应抹掉这部分新增。

## 上传流程

### 上传前

队列中的标签编辑只存在于前端本地状态，不提前调用保存接口，也不增加额外的“单图保存”动作。

### 上传中

点击“上传全部”后，仍按现有逐张上传流程处理，但每张图片在发请求前都先通过统一 helper 计算自己的最终标签集合。

上传请求中：

- `tagIds` 改为当前图片自己的最终 `effectiveTagIds`
- `tagNames` 改为当前图片自己的最终 `effectiveTagNames`

不再直接对所有图片复用顶部 `defaultTagIds` 和 `newTagNames`。

### 重试

如果单张图片上传失败：

- 保留该图片的单图标签覆盖状态
- 允许用户继续编辑或直接重试
- 重试时继续按当前最新的最终标签集合提交

## 状态限制

为了避免用户在请求已发出后继续修改本次上传的 payload：

- `ready` 和 `failed` 状态允许编辑标签
- `signing`、`uploading`、`saving`、`complete` 状态禁止编辑标签

这与现有删除按钮的可用范围保持一致，也更符合用户对“待上传队列”的直觉。

## 错误处理

### 前端

- 标签输入统一去重、去首尾空格、忽略空值。
- 队列项的单图编辑只改本地状态，不产生新的后端失败点。
- 上传失败时保留当前编辑状态，不清空用户已做的单图调整。

### 后端

上传接口预计无需修改协议，只需继续接收：

- `tagIds`
- `tagNames`

前端本次只调整这两个字段的来源。

## 测试策略

### 纯函数测试

新增标签结果计算 helper 后，优先围绕它补单元测试，至少覆盖：

- 顶部默认标签会体现在队列项最终结果里
- 单图可以额外新增标签
- 单图可以取消某个批量默认标签
- 顶部删除默认标签后，队列项最终结果同步消失
- 名称标签会去重、去空白、忽略空值

### 组件行为测试

围绕上传组件补最小必要的行为测试，至少覆盖：

- 顶部标签切换后，队列项标签状态实时变化
- 单图编辑后，再改顶部标签，不会丢失单图覆盖
- 上传请求发送的是单图最终标签，而不是纯顶部默认标签

不在这次引入更重的端到端测试，保持改动聚焦。

## 实施边界

主要改动文件预计为：

- `src/components/upload-dropzone.tsx`

如果为了可测试性需要抽离 helper，则允许新增一个很小的标签计算模块及对应测试文件。

后端上传路由预计不需要结构性改动，除非实现过程中发现现有字段校验与单图标签结果不兼容。

## 风险与应对

### 风险 1：顶部与单图状态互相覆盖

应对：

- 不在队列项里保存完整标签快照
- 统一通过“默认值 + 覆盖层”计算最终结果

### 风险 2：UI 看得见的标签与实际上传标签不一致

应对：

- 渲染和上传共用同一个最终标签计算 helper

### 风险 3：新标签名的字符串比较不稳定

应对：

- 统一在输入解析阶段做 `trim`、去空值、去重
- 内部始终使用解析后的数组参与比较和计算
