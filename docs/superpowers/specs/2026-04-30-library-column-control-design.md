# 图库列数控制设计

## 目标

在图库主页面左上角新增一个悬浮列数控制器，用来实时调整图片瀑布流的列数。

本次设计需要满足：

- 控件位于图库页左上角，视觉语言与右上角现有搜索筛选栏保持一致
- 默认折叠，只显示一个 8x8 的圆形悬浮按钮
- 点击后展开一个横向滑轨，支持左右拖动调节列数
- 列数范围固定为 3 到 8，步进为 1
- 默认列数为 4
- 调节时图片网格实时更新
- 展开与折叠动画沿用右上角搜索筛选栏当前的动画节奏和缓动
- 点击空白区域或按 `Esc` 可以收起
- 现有搜索与标签筛选逻辑保持不变，不与列数状态互相影响
- 不持久化到 URL、数据库或本地存储，刷新后恢复默认值 4

## 当前状态

当前图库页由 `src/app/dashboard/library/page.tsx` 作为服务端入口，负责：

- 读取搜索参数
- 查询图片和标签
- 渲染 `LibraryFilterBar`
- 渲染 `ImageGrid`

当前右上角筛选栏 `src/components/library-filter-bar.tsx` 已经具备：

- 折叠圆形按钮
- 展开后的模糊悬浮容器
- `AnimatePresence` 切换
- 点击外部关闭
- `Esc` 关闭
- 固定的动画时间与 easing

当前 `src/components/image-grid.tsx` 仍然把列数写死在类名里：

- `columns-2 md:columns-3 xl:columns-4`

这意味着图库列数无法由交互状态驱动，也无法与新的左上角列数控制器连接。

## 核心决策

### 1. 保留服务端数据加载，新增客户端外壳

`src/app/dashboard/library/page.tsx` 继续只负责服务端取数和数据整形，不在这里引入交互状态。

新增一个轻量客户端容器组件，负责：

- 承载左上角列数控制器
- 承载右上角现有搜索筛选栏
- 在客户端维护 `columnCount`
- 把 `columnCount` 透传给 `ImageGrid`

这样可以把页面数据读取与界面交互解耦，避免把现有 server component 改成 client component。

### 2. 列数状态只存在于客户端

列数状态定义为客户端临时 UI 状态：

- 默认值：`4`
- 最小值：`3`
- 最大值：`8`
- 输入统一做 clamp，防止未来调用方传入越界值

本次不做：

- URL 参数同步
- `localStorage` 记忆
- 用户级持久化配置

刷新页面后恢复到 4，是刻意保留的简单行为，不额外引入状态同步复杂度。

### 3. 左上角控件复用右上角筛选栏的交互模型

新的列数控制器不重新发明一套展开逻辑，而是沿用 `LibraryFilterBar` 已有模式：

- 折叠态：圆形图标按钮
- 展开态：模糊背景的横向悬浮条
- `AnimatePresence` 做切换
- 折叠按钮使用 `opacity + scale`
- 展开容器使用 `opacity + scaleX + scaleY`
- 时间与 easing 与现有筛选栏保持一致

需要保留一个关键差异：

- 右上角筛选栏展开原点是 `right`
- 左上角列数栏展开原点应为 `left`

这样两个控件会从各自贴边的一侧展开，视觉上统一，但方向正确。

### 4. `ImageGrid` 改为接收受控列数

`ImageGrid` 新增 `columnCount` 属性，不再内部写死响应式列类。

实现上以受控列数为唯一来源，按列数生成内联样式或等价的受控 class，确保：

- 调节滑块时立即生效
- 列数不会再被 `md/xl` 固定断点覆盖
- 页面上的图片布局由单一状态驱动

这里的目标是“可控列数的瀑布流”，不是“响应式断点自动列数”。
因此本次以用户当前选择为准，而不是继续混合固定断点类名。

### 5. 两个悬浮控件互相独立

左上角列数栏与右上角搜索筛选栏应各自维护展开状态，互不干扰：

- 打开列数栏不会重置搜索词
- 打开筛选栏不会重置列数
- 一侧收起不会强制另一侧同步收起

这两个控件共享视觉语言，但不共享业务状态。

## 组件边界

### `DashboardLibraryPage`

保留服务端职责：

- 解析 `q` 和 `tag`
- 查询图片与标签
- 读取 OSS 配置
- 把数据传给新的客户端容器

### 新增客户端容器

建议新增例如 `src/components/library-page-shell.tsx` 一类的客户端组件，职责为：

- 管理 `columnCount`
- 渲染左侧列数控件
- 渲染右侧 `LibraryFilterBar`
- 渲染 `ImageGrid`

该组件不负责取数，只消费服务端已经准备好的 props。

### 新增列数控件

建议新增例如 `src/components/library-column-control.tsx`，职责为：

- 管理自身展开/收起
- 渲染折叠按钮与展开滑轨
- 响应滑块输入
- 对外输出合法列数

### `ImageGrid`

保留图片渲染、点击跳转、空状态展示等职责，但不再决定列数策略。

## 交互设计

### 折叠态

- 控件固定在图库内容区域顶部左侧
- 外观与右上角搜索按钮同级：小圆形、半透明、模糊背景、轻量描边
- 图标可使用网格相关图标，表达“调整布局”

### 展开态

- 展开后显示横向轨道和当前数值
- 滑块范围 `3..8`
- 步长 `1`
- 当前列数应直接可见，避免用户拖动时需要猜测

### 操作反馈

- 拖动滑块时网格即时响应
- 数值变更后不需要额外确认按钮
- 点击空白或按 `Esc` 收起时，当前值保留在本次页面生命周期内

### 空图库场景

即使当前图库为空，也允许打开列数控件。
这时控件仍然正常展示和修改列数，只是暂时没有图片可见变化。

## 动画与样式

列数控件的动画参数应直接对齐 `LibraryFilterBar` 当前配置：

- 折叠按钮进入/退出：`duration: 0.15`, `ease: "easeOut"`
- 展开容器进入/退出：`duration: 0.2`, `ease: [0.4, 0, 0.2, 1]`

视觉元素保持同一套语言：

- 圆角
- 半透明黑色背景
- `backdrop-blur-md`
- 轻量 ring
- hover 反馈

需要明确的是：本次是“样式和动画语言复用”，不是抽象成一个通用浮动栏基类。除非实现时发现明显重复，否则不额外做抽象。

## 数据与状态流

1. 服务端页面读取搜索参数并查询图片数据
2. 服务端页面把 `images`、`tags`、`query`、`selectedTagIds` 传给客户端容器
3. 客户端容器初始化 `columnCount = 4`
4. 列数控件通过滑块更新 `columnCount`
5. `ImageGrid` 接收新的 `columnCount` 并重新渲染列布局
6. 搜索与标签筛选继续通过 URL 刷新服务端数据，与列数状态并行存在

## 错误处理与边界

- 任何来自控件或调用方的列数值都应 clamp 到 `3..8`
- 如果未来 `ImageGrid` 在未传入 `columnCount` 的情况下被复用，组件内部应有安全默认值 `4`
- 搜索、标签筛选、空状态、图片点击跳转等现有行为不得回归
- 不要求在移动端做单独列数规则，移动端同样遵循 `3..8` 的统一约束

## 测试策略

本次以契约和状态逻辑测试为主，不要求重型端到端覆盖。

建议覆盖：

- `ImageGrid` 默认列数为 `4`
- `ImageGrid` 接收受控 `columnCount` 后按指定列数渲染
- 列数输入会被 clamp 到 `3..8`
- 新控件默认折叠，展开后存在 `3..8` 滑块与当前数值
- 新控件保留与 `LibraryFilterBar` 一致的动画模式，且展开原点为 `left`
- 点击外部和 `Esc` 可收起，不丢失当前页面内已选择的列数

## 实施边界

主要改动范围：

- `src/app/dashboard/library/page.tsx`
- 新增图库客户端容器组件
- 新增列数控制组件
- `src/components/image-grid.tsx`
- 相关测试

本次不做：

- 列数偏好持久化
- URL 同步列数
- 重做图库搜索栏
- 重写图片瀑布流算法
- 针对不同屏宽做独立列数策略
